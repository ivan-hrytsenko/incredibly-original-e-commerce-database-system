# Лабораторна роботи №5

## Аналіз початкової схеми
Початкова схема бази даних (створена в Лабораторній роботі №2) була перевірена на відповідність нормальним формам:
* **1NF:** Виконується. Усі атрибути атомарні, повторюваних груп немає.
* **2NF:** Виконується. Усі таблиці мають прості первинні ключі (одне поле), тому часткові залежності відсутні.
* **3NF:** **Порушується** у таблиці `addresses`.

## Виявлення проблеми (Транзитивна залежність)
У таблиці `addresses` було виявлено таку функціональну залежність (FD):
* `address_id` (PK) -> `zip_code`
* `zip_code` -> `city`

Оскільки `city` залежить від `address_id` не безпосередньо, а через `zip_code` (інший неключовий атрибут), ми маємо транзитивну залежність. Це спричиняє надмірність: якщо 100 клієнтів живуть в одному місті, назва міста дублюється 100 разів.

## 3. Процес нормалізації

### Декомпозиція
Для усунення залежності `zip_code -> city` була створена нова таблиця-довідник `zip_codes`.

**Нова структура:**
* **zip_codes** (`zip_code` [PK], `city`)
* **addresses** (`address_id` [PK], `street`, `house_number`, `zip_code` [FK])

### Забезпечення цілісності
Стовпець `city` було видалено з таблиці `addresses`. Зв'язок між таблицями реалізовано через зовнішній ключ (Foreign Key) по полю `zip_code`.

## Фінальний дизайн (3NF)
Після проведених змін кожен неключовий атрибут у схемі залежить **тільки від первинного ключа**. Це мінімізує ризик виникнення аномалій оновлення та видалення даних.
